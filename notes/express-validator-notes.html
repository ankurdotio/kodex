express-validator

A battle-tested middleware library for validating and sanitizing user input in Express.js applications — built on top of the powerful validator.js library.

v7.x express.js validator.js middleware REST APIs
01
Introduction
Why we validate, and what express-validator gives us
Key Concept

Validation checks that data is correct in type and format (e.g. is this a valid email?). Sanitization cleans and transforms data (e.g. trim whitespace, escape HTML). Both are essential before touching a database.

When users submit a form or call your API, you never trust the incoming data. A missing field, an invalid email, or a malformed ID can crash your server, corrupt your database, or open security holes. express-validator solves all of this elegantly by turning validation logic into reusable Express middleware.

Why not just write if-checks manually?

AspectManual if-checksexpress-validator
Readability Nested conditionals, hard to scan Declarative chain reads like English ✓
Reusability Copy-paste across routes Extract into a shared array of middleware ✓
Error collection You build it yourself Built-in validationResult() ✓
Sanitization Additional utility functions needed Built right in — trim, escape, normalise ✓
Async checks Messy promise handling Native async custom validators ✓
💡

express-validator wraps validator.js — so every string validation method you know from validator.js (isEmail, isURL, isUUID, etc.) is available directly on the chain.

02
Installation & Setup
Getting the package into your project
Step 1 — Install
Terminal 
npm install express-validator
Step 2 — Import what you need
app.js / routes/auth.js 
const { body, param, query, check, validationResult } = require('express-validator');

// ES Module (if using "type": "module" in package.json)
import { body, param, query, check, validationResult } from 'express-validator';
Named Exports — What each one targets

body('field') — targets req.body.field  |  param('id') — targets req.params.id  |  query('page') — targets req.query.page  |  check('field') — searches body, params, query, cookies, and headers all at once.

Verify — Minimal working setup
server.js — minimal example 
const express = require('express');
const { body, validationResult } = require('express-validator');

const app = express();
app.use(express.json());     // ← must come BEFORE routes, parses req.body

app.post('/register',
  body('email').isEmail(),        // validation middleware
  (req, res) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      return res.status(422).json({ errors: errors.array() });
    }
    res.send('Registered!');
  }
);

app.listen(3000);
⚠️

Always add express.json() before your routes. Without it, req.body is undefined and every body() validator will silently fail or treat the field as missing.

03
check() and body()
The two most common ways to start a validation chain

Both check() and body() return a Validation Chain — an Express middleware that, when called, runs all attached validators and stores any errors on the request object. You chain methods onto it fluently.

Anatomy of a Validation Chain

Example 
body('username')          // 1. which field?
  .trim()               // 2. sanitize first
  .notEmpty()           // 3. must not be blank
  .isLength({ min: 3, max: 20 })  // 4. length check
  .withMessage('Username must be 3–20 chars')  // 5. custom error message

Flow of a request

Request hits route
POST /register with JSON body arrives at Express
Validation middleware runs
Each body() / check() call is middleware — they run in order, collect errors on req
Route handler calls validationResult(req)
Extracts all collected errors from the request object
Respond based on result
If errors → 422 with error list. If clean → process the data.

check() vs body() — when to use which

FunctionSearches inBest for
body('field') req.body only ✓ Form/JSON payloads — most common ✓
param('id') req.params only ✓ URL segments like /users/:id ✓
query('page') req.query only ✓ ?page=2&sort=asc style params ✓
check('field') body, params, query, cookies, headers Broad check — less explicit, avoid in large codebases
04
Built-in Validators
The full set of checking methods available on the chain

Every method below is chained onto body(), param(), or query(). They return the same chain, so you can keep adding checks.

Existence & Emptiness

.exists()
Field must exist on the request (even if empty string). Fails only if undefined.
.notEmpty()
Field must not be an empty string after trimming. Recommended over not().isEmpty().
.optional()
Skips all validators if the field is absent or undefined. Use to make a field truly optional.
.optional({ nullable: true })
Skips validation if field is absent or null. Useful when frontend can send null explicitly.
Existence examples 
body('name').notEmpty().withMessage('Name is required')

body('bio').optional().isLength({ max: 200 })  // only checks length if bio is present

body('avatar').optional({ nullable: true }).isURL()  // allows null or missing

String Validators

MethodOptionsWhat it validates
.isEmail() — Valid email address format (uses validator.js isEmail)
.isURL() { protocols, require_tld } Valid URL. Accepts options to restrict protocols (http, https)
.isLength() { min, max } String length between min and max (inclusive)
.isAlpha() locale Contains only letters A-Z. Optional locale for unicode.
.isAlphanumeric() locale Contains only letters and numbers
.isStrongPassword() { minLength, minNumbers, minSymbols } Password strength: uppercase, lowercase, numbers, symbols
.matches() /regex/ Must match the provided regular expression
.isIn() ['a','b','c'] Value must be one of the allowed values
.contains() 'seed' String must contain the given seed string

Number Validators

MethodOptionsWhat it validates
.isInt() { min, max } Is an integer. Optional min/max range constraints.
.isFloat() { min, max } Is a floating point number, with optional range
.isNumeric() — Contains only numeric characters (allows sign and decimal)
.isDecimal() { decimal_digits } Is a decimal number with optional precision settings

Format / Special Validators

MethodWhat it validates
.isMongoId() Valid MongoDB ObjectID (24-char hex string)
.isUUID() Valid UUID (v1, v2, v3, v4, v5, or "all")
.isDate() Parseable date string (e.g. 2024-01-15)
.isISO8601() ISO 8601 datetime format (e.g. 2024-01-15T10:30:00Z)
.isBoolean() Value is "true", "false", "1", or "0"
.isIP() Valid IPv4 or IPv6 address
.isJWT() Valid JWT format (three dot-separated base64 segments)
.isJSON() Valid JSON string (parseable)
.isMobilePhone() Valid phone number, supports locale ('en-IN', 'any', etc.)
.isPostalCode() Valid postal/zip code for a given locale
.isCreditCard() Valid credit card number (Luhn algorithm check)

Practical Example — User Registration

routes/auth.js 
const registerValidators = [
  body('name')
    .trim()
    .notEmpty().withMessage('Name is required')
    .isLength({ min: 2, max: 50 }).withMessage('Name must be 2–50 characters'),

  body('email')
    .normalizeEmail()       // sanitize: lowercase, remove dots in Gmail
    .isEmail().withMessage('Enter a valid email'),

  body('password')
    .isStrongPassword({
      minLength: 8,
      minNumbers: 1,
      minSymbols: 1,
    })
    .withMessage('Password needs 8+ chars, 1 number, 1 symbol'),

  body('age')
    .optional()
    .isInt({ min: 13, max: 120 }).withMessage('Age must be between 13 and 120'),
];

router.post('/register', registerValidators, registerController);
05
Sanitizers
Clean and transform data before it reaches your database
Key Concept

Sanitizers modify req.body in place. After sanitizer middleware runs, the field value on req.body.field is the cleaned version — not the original. Always sanitize before validation for most accurate results.

Common Sanitizers

MethodWhat it doesExample
.trim() Remove leading/trailing whitespace " hello " → "hello"
.escape() Convert HTML special chars to entities — prevents XSS "<b>" → "&lt;b&gt;"
.toLowerCase() Convert string to lowercase "HELLO" → "hello"
.toUpperCase() Convert string to uppercase "hello" → "HELLO"
.normalizeEmail() Lowercase + remove Gmail dots + handle aliases "Foo.Bar@Gmail.com" → "foobar@gmail.com"
.toInt() Convert string to integer "42" → 42 (number)
.toFloat() Convert string to float "3.14" → 3.14 (number)
.toBoolean() Convert "true"/"1" to true, "false"/"0" to false "true" → true
.stripLow() Remove low ASCII control characters Removes \x00–\x1F chars
.blacklist(chars) Remove specified characters from the string blacklist('0-9') removes all digits
.whitelist(chars) Keep only the specified characters whitelist('a-z') keeps only lowercase
Best Practice — Sanitize order
Sanitize THEN validate 
// ✅ Correct order: sanitize first, then validate
body('username')
  .trim()                    // 1. remove whitespace
  .toLowerCase()             // 2. normalize case
  .notEmpty()                // 3. now check it's not empty
  .isAlphanumeric()          // 4. check format on clean value
  .isLength({ min: 3, max: 20 })

// ❌ If you validate before trim, " " passes notEmpty() but is invalid
💡

After sanitizers run, req.body.username is already the cleaned value in your controller. You don't need to trim/lowercase again — it's done. This is one of the big wins over manual validation.

06
validationResult()
Collecting and responding to errors from the request

validationResult(req) collects all errors that were attached to req by the validation middleware. It returns a Result object — not an array directly.

The Result Object

MethodReturnsUse case
.isEmpty() boolean true if there are NO errors. Most common check.
.array() Error[] All errors as an array. Best for sending in API response.
.mapped() { field: Error } Errors as an object keyed by field name. Only the first error per field.
.throw() throws Throws an error if any validation failed. Use in try/catch flows.
.formatWith(fn) Result Custom error formatter before calling .array()

Shape of each error object

Error object shape (from .array()) 
{
  type:     "field",           // error type
  value:    "",               // the invalid value that was submitted
  msg:      "Invalid email",  // your withMessage() string
  path:     "email",          // the field name
  location: "body"            // "body", "params", "query", etc.
}

Three patterns for handling errors

Pattern 1 — Return errors inline (most common)
controller.js 
const registerUser = async (req, res) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({
      success: false,
      errors: errors.array(),
    });
  }
  // safe to use req.body here
};
Pattern 2 — mapped() for field-level errors (great for forms)
controller.js — field map 
const errors = validationResult(req).mapped();
// { email: { msg: 'Invalid email', ... }, name: { msg: '...' } }

if (Object.keys(errors).length > 0) {
  return res.status(422).json({ errors });
}
Pattern 3 — Extracting a reusable middleware
middleware/validate.js — shared error handler 
// Define once, use everywhere
const validate = (req, res, next) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({ errors: errors.array() });
  }
  next();
};

// In your route:
router.post('/register',
  [...registerValidators],   // validation chains
  validate,                  // shared error checker ← clean!
  registerController         // only runs if validation passed
);
💡

Pattern 3 is strongly recommended for production code. It keeps controllers clean and DRY — no copy-pasting the error check into every handler.

07
Chaining & withMessage()
Multiple validators, per-check messages, and negation

withMessage() — per validator messages

By default, failed validators use a generic error message from validator.js. You almost always want to override this.

Multiple validators, individual messages 
body('password')
  .notEmpty().withMessage('Password is required')
  .isLength({ min: 8 }).withMessage('Minimum 8 characters')
  .matches(/\d/).withMessage('Must contain at least one digit')
  .matches(/[!@#$%]/).withMessage('Must contain a special character')
💡

Each .withMessage() applies to the validator immediately before it, not the whole chain. If you put .withMessage() at the end, it only covers the last check.

Negation with .not()

Prefix any validator with .not() to invert it.

Negation examples 
body('username')
  .not().isIn(['admin', 'root', 'superuser'])
  .withMessage('This username is reserved')

body('email')
  .not().contains('+')
  .withMessage('Email aliases with + are not allowed')

bail() — stop on first failure

By default all validators in a chain run even if an early one fails. .bail() stops the chain as soon as the previous check fails — saves unnecessary work.

bail() usage 
body('email')
  .notEmpty().bail()       // if empty, stop here — don't run isEmail
  .isEmail().bail()         // if not email, stop — don't run DB check
  .custom(async (email) => {
    const user = await User.findOne({ email });
    if (user) throw new Error('Email already registered');
  });
08
Schema Validation
Define all validators as a plain object — great for complex forms
Key Concept

checkSchema() accepts a single object where keys are field names and values describe the full validation config for that field. Avoids long arrays of body() calls — especially clean for API documentation purposes.

checkSchema() example 
const { checkSchema } = require('express-validator');

const createUserSchema = {
  name: {
    trim: true,
    notEmpty: { errorMessage: 'Name is required' },
    isLength: { options: { min: 2, max: 50 }, errorMessage: '2-50 chars' },
  },
  email: {
    normalizeEmail: true,
    isEmail: { errorMessage: 'Enter a valid email' },
  },
  age: {
    optional: true,
    isInt: { options: { min: 13 }, errorMessage: 'Must be 13 or older' },
    toInt: true,
  },
};

router.post('/user', checkSchema(createUserSchema), validate, createUser);
💡

Schema style is preferred in teams because it's easy to read, self-documenting, and can be stored in a separate file (e.g. schemas/createUser.js) keeping your routes and controllers clean.

09
Route Integration
Wiring validators into a real Express router structure

Here's a complete, production-style example showing how to structure your route files with validators extracted into their own module.

File 1 — validators/userValidator.js
validators/userValidator.js 
const { body, param } = require('express-validator');

const createUserValidators = [
  body('name').trim().notEmpty().isLength({ min: 2 }),
  body('email').normalizeEmail().isEmail(),
  body('password').isStrongPassword().withMessage('Weak password'),
];

const updateUserValidators = [
  param('id').isMongoId().withMessage('Invalid user ID'),
  body('name').optional().trim().isLength({ min: 2 }),
  body('email').optional().normalizeEmail().isEmail(),
];

module.exports = { createUserValidators, updateUserValidators };
File 2 — middleware/validate.js
middleware/validate.js 
const { validationResult } = require('express-validator');

const validate = (req, res, next) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(422).json({
      success: false,
      message: 'Validation failed',
      errors: errors.array().map(e => ({ field: e.path, msg: e.msg })),
    });
  }
  next();
};

module.exports = validate;
File 3 — routes/users.js
routes/users.js 
const router = require('express').Router();
const { createUserValidators, updateUserValidators } = require('../validators/userValidator');
const validate = require('../middleware/validate');
const userController = require('../controllers/userController');

router.post('/',       createUserValidators, validate, userController.create);
router.put('/:id',    updateUserValidators, validate, userController.update);
router.delete('/:id',
  [param('id').isMongoId()], validate, userController.remove
);

module.exports = router;
Request arrives
    ↓
Validation chains run (body, param, query)
    ↓
validate middleware checks result
    ↓
Errors? → 422 with errors array
    ↓
Clean? → next() → Controller runs ✅

Conditional Validation
Validate fields only when certain conditions are met

if() — run a check only when a condition passes

Conditional — validate phone only if role is 'driver' 
body('phone')
  .if(body('role').equals('driver'))
  .notEmpty().withMessage('Phone is required for drivers')
  .isMobilePhone('en-IN')

unless() — run a check unless a condition passes

Unless — require address unless user is "guest" 
body('address')
  .unless(body('userType').equals('guest'))
  .notEmpty().withMessage('Address required for registered users')

Validating arrays of fields

Array validation — each item in req.body.tags 
// req.body.tags = ['javascript', 'nodejs', ...]
body('tags')
  .isArray({ min: 1, max: 5 })
  .withMessage('Provide 1–5 tags')

body('tags.*')             // wildcard — validates EACH item in the array
  .trim()
  .isString()
  .isLength({ min: 2, max: 20 })
  .withMessage('Each tag must be 2–20 characters')

Nested objects

Nested — req.body.address.city 
// req.body = { address: { city: 'Mumbai', pincode: '400001' } }
body('address.city').notEmpty().isAlpha()
body('address.pincode').isPostalCode('IN')
11
Custom Validators
Write your own logic — including async database checks
Key Concept

.custom(fn) accepts a function that receives the field value and a meta object { req, path, location }. To fail validation, throw an Error or return a rejected Promise. To pass, return nothing or a resolved Promise.

Synchronous custom validator

custom() — sync 
body('username')
  .custom((value) => {
    if (value.includes(' ')) {
      throw new Error('Username cannot contain spaces');
    }
    return true;  // explicitly return true to pass
  })

Async custom validator — check database uniqueness

custom() — async DB check 
body('email')
  .isEmail().bail()           // bail so DB isn't hit with invalid email
  .custom(async (email) => {
    const existingUser = await User.findOne({ email });
    if (existingUser) {
      throw new Error('Email is already registered');
    }
  })

Cross-field validation — password confirm

Access other fields via meta.req 
body('confirmPassword')
  .custom((value, { req }) => {
    if (value !== req.body.password) {
      throw new Error('Passwords do not match');
    }
    return true;
  })
⚠️

Always use .bail() before async custom validators. If isEmail() fails, you don't want to fire a DB query with a garbage email — that wastes resources and can cause Mongoose errors.

12
Error Handling Patterns
Clean, consistent API error responses

Standard API error response shape

Keep a consistent JSON shape across all your endpoints so frontend can handle errors uniformly.

Response shape 
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    { "field": "email",    "msg": "Enter a valid email" },
    { "field": "password", "msg": "Minimum 8 characters" }
  ]
}

Production-grade validate middleware

middleware/validate.js — full version 
const { validationResult } = require('express-validator');

const validate = (req, res, next) => {
  const result = validationResult(req);

  if (result.isEmpty()) return next();

  const errors = result.array().map(({ path, msg }) => ({ field: path, msg }));

  return res.status(422).json({
    success: false,
    message: 'Validation failed',
    errors,
  });
};

module.exports = validate;

Quick Reference — Which HTTP status to use

SituationAvoidUse instead
Validation failed (bad input format) 400 Bad Request 422 Unprocessable Entity ✓
Resource not found by valid ID 422 404 Not Found ✓
Unauthorised action 422 403 Forbidden ✓
Missing auth token 422 401 Unauthorised ✓

Complete working example — User Registration route

routes/auth.js — full register route 
const express  = require('express');
const bcrypt   = require('bcrypt');
const User     = require('../models/User');
const validate = require('../middleware/validate');
const { body } = require('express-validator');

const router = express.Router();

const registerValidators = [
  body('name')
    .trim()
    .notEmpty().withMessage('Name is required')
    .isLength({ min: 2, max: 50 }).withMessage('2–50 characters only'),

  body('email')
    .normalizeEmail()
    .isEmail().withMessage('Enter a valid email').bail()
    .custom(async (email) => {
      if (await User.findOne({ email })) throw new Error('Already registered');
    }),

  body('password')
    .isStrongPassword({ minLength: 8, minNumbers: 1, minSymbols: 1 })
    .withMessage('8+ chars, 1 number, 1 symbol required'),

  body('confirmPassword')
    .custom((val, { req }) => {
      if (val !== req.body.password) throw new Error('Passwords do not match');
      return true;
    }),
];

router.post('/register', registerValidators, validate, async (req, res) => {
  const { name, email, password } = req.body;  // already sanitized
  const hashed = await bcrypt.hash(password, 10);
  const user = await User.create({ name, email, password: hashed });
  res.status(201).json({ success: true, userId: user._id });
});

module.exports = router;
💡

Because sanitizers ran before the controller, req.body.email is already normalized (lowercased, Gmail dots removed). You save it directly — no extra processing needed.
